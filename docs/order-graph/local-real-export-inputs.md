# Local Real-Export Input Contracts

## Why This File Exists

This file defines what local real-shaped export data must look like before ChaOS is allowed to ingest it.

The Order Graph has proven its first local fixture path. The next risk is not whether the graph can process clean fake data. The next risk is whether it can accept messy exported business data without guessing, leaking sensitive information, or treating context as hard truth.

This document keeps the next implementation step documentation-first. It defines input contracts, field boundaries, sanitization rules, and approved local folder structure before any import adapter is added.

## Purpose

The purpose of this contract is to answer one question:

```text
What does real data need to look like before ChaOS is allowed to ingest it?
```

The answer must be specific enough for future import code to follow, but simple enough for a human reviewer to audit.

## How This Fits Into ChaOS

This file supports the transition from fake GTM fixtures to safe, real-shaped local inputs.

It follows the ChaOS principles that:

- Context is a dependency. Minimize it.
- Systems must be understandable by humans first.
- Recommend before automating.
- Prefer inspectable systems over intelligent systems.
- Every decision must be traceable.

This contract does not approve live integrations, production data ingestion, CRM writes, outreach actions, agents, UI, deployment, or autonomous behavior.

## Scope

This contract applies only to local export files prepared for Order Graph testing.

Approved local export inputs may include:

- CSV files.
- JSON files.
- Local mapping configuration files.
- Sanitized fake-but-real-shaped sample files.

Approved local export inputs must not include:

- Live API connections.
- Production credentials.
- Raw customer secrets.
- Private notes that have not been classified.
- Personal data that is not needed for the test.
- Files generated by an autonomous agent without human review.

## Required Export Types

The first local real-export test must include enough data to create SourceRecords for core GTM identity and signal behavior.

### Account Or Company Export

Purpose: represent organizations that may resolve into Account entities.

Minimum fields:

- Source system name.
- Source record identifier.
- Account or company name.
- Domain or website when available.
- Owner or responsible team when available.
- Created date or first-seen date when available.
- Last updated date when available.

Optional fields:

- Industry.
- Employee count band.
- Region.
- Account status.
- Parent account reference.
- External account reference.

### Person Or Contact Export

Purpose: represent people who may resolve into Contact entities.

Minimum fields:

- Source system name.
- Source record identifier.
- Email address when available.
- First name and last name when available.
- Associated account, company, domain, or source account identifier when available.
- Created date or first-seen date when available.
- Last updated date when available.

Optional fields:

- Title.
- Role.
- Seniority.
- Department.
- Owner or responsible team.
- Linked source account identifier.

### Activity Or Signal Export

Purpose: represent time-based evidence that may become Signal objects.

Minimum fields:

- Source system name.
- Source record identifier.
- Signal or activity type.
- Event time.
- Actor reference when available.
- Account, company, domain, opportunity, sequence, campaign, or workflow reference when available.
- Event summary or structured event label.

Optional fields:

- Direction.
- Channel.
- Outcome label.
- Confidence label from source system.
- Workflow or campaign identifier.
- Related opportunity identifier.

### Opportunity Or Commercial Object Export

Purpose: represent commercial context when available without making opportunity state required for the first real-export test.

Minimum fields when included:

- Source system name.
- Source record identifier.
- Opportunity or deal name.
- Associated account identifier.
- Stage or status.
- Created date when available.
- Last updated date when available.

Optional fields:

- Amount band.
- Close date.
- Owner.
- Loss reason.
- Renewal date.
- Forecast category.

Opportunity data may help explain state, but the first real-export ingestion path must not require it.

## Minimum File-Level Requirements

Every local export file must have:

- A clear source system label.
- A clear object type label.
- Stable column names or JSON field names.
- A corresponding mapping configuration.
- A corresponding field classification configuration.
- A reviewer-readable import purpose.

Every local export file must be small enough for human inspection during early tests.

The first real-shaped local test should target:

- 25 to 50 accounts.
- 50 to 100 people or activity rows.
- Enough messy records to prove unresolved handling.

## Field Classification Rules

Every field must be classified before ingestion.

The field classification model will be formalized in a later document, but this contract establishes the first required classes.

### Structured Truth

Structured truth fields may be used as hard evidence when the source field is stable and explicit.

Examples:

- Source record identifiers.
- Source system names.
- Domains.
- Email addresses.
- Dates.
- Owner identifiers.
- Explicit account identifiers.
- Explicit contact identifiers.
- Explicit opportunity identifiers.
- Source-native workflow identifiers.

Structured truth fields may support identity resolution, relationship linking, signal attachment, and traceability.

### Contextual Evidence

Contextual evidence fields may help explain a recommendation, but they must not be treated as identity truth or automatic state truth.

Examples:

- Notes.
- Descriptions.
- Call summaries.
- Email snippets.
- Free-text next steps.
- Human-entered status comments.

Contextual evidence must remain labeled as context unless a future reviewed rule promotes a specific structured extraction.

### Derived Signal

Derived signal fields may represent a source-system interpretation.

Examples:

- Intent score band.
- Engagement category.
- Health label.
- Forecast category.
- Risk label.

Derived signal fields may be preserved, but ChaOS must not treat them as hidden authority. The originating source and derivation risk must remain visible.

### Restricted

Restricted fields may exist in a source export but must not be ingested into the early local pipeline.

Examples:

- Sensitive free text.
- Private customer notes.
- Legal notes.
- HR-like fields.
- Unredacted phone numbers.
- Personal addresses.
- Internal-only commentary.

Restricted fields must be excluded from generated SourceRecords unless a future decision record approves a specific safe handling pattern.

### Excluded

Excluded fields must not be copied, transformed, summarized, or preserved.

Examples:

- Credentials.
- Tokens.
- API keys.
- Session identifiers.
- Payment information.
- Social security numbers or government identifiers.
- Raw files that contain unreviewed sensitive data.

Excluded fields must fail review if they appear in an approved local test folder.

### Unknown Needs Review

Unknown fields must not be ingested until a human classifies them.

An import adapter must preserve unknown field names in warnings without preserving sensitive values.

## Structured Truth Versus Contextual Evidence

ChaOS must separate structured truth from contextual evidence.

Structured truth can answer:

- What source record is this?
- Which source system did it come from?
- Which explicit identifier connects this record to another record?
- When did this happen?
- Which domain or email is present?

Contextual evidence can answer:

- What might this mean?
- What did a human or source system say about it?
- What uncertainty should a reviewer see?
- What context might explain a recommendation?

Contextual evidence must not silently create identity matches, high-priority state, or recommendations.

## Sanitization Rules

Local real-shaped test data must be sanitized before it is committed to the repository.

Sanitization must include:

- Replace real company names with fake company names unless the data is already public and approved for use.
- Replace real person names with fake names.
- Replace real email addresses with fake domains.
- Remove phone numbers unless they are fake.
- Remove postal addresses.
- Remove credentials, tokens, keys, and session identifiers.
- Remove sensitive free-text notes.
- Preserve shape, messiness, and field behavior without preserving private content.

Sanitized data may preserve:

- Relative dates.
- Missing fields.
- Duplicate-like records.
- Conflicting source values.
- Stale activity patterns.
- Unknown fields by name.
- Suppression-like flags when values are safe.

The goal is to preserve the shape of ugly data, not the private content of real data.

## Approved Local Test Folder Structure

Committed fake-but-real-shaped examples may use:

```text
examples/order-graph/local-real/
```

Future committed examples may include:

```text
examples/order-graph/local-real/README.md
examples/order-graph/local-real/exports/
examples/order-graph/local-real/expected-source-records.json
```

Committed mapping examples may use:

```text
configs/order-graph/local-real/
```

Future committed mapping examples may include:

```text
configs/order-graph/local-real/field-mapping.example.json
configs/order-graph/local-real/field-classification.example.json
```

Uncommitted local real exports must remain outside the repository unless they have been sanitized and approved.

Recommended uncommitted local folder:

```text
local-data/order-graph/exports/
```

The `local-data/` folder must be ignored if it is introduced. Real exports must not be committed by accident.

## Import Readiness Checklist

A local export is ready for future import adapter testing only when:

- Required export types are present or explicitly waived.
- Required minimum fields are present or missing fields are documented.
- Every field has a classification.
- Restricted fields are excluded from ingestion.
- Excluded fields are absent.
- Unknown fields are listed for review without sensitive values.
- Source system and object type are clear.
- Mapping configuration exists.
- Sanitization has been reviewed.
- The test remains small enough for a human reviewer to inspect.

## Examples Of Use

A reviewer may inspect a proposed local real-shaped export and ask:

- Does this account export include source identifiers, names, and domains where available?
- Does this contact export include email or another explicit source identifier?
- Does this activity export include event time and source record identifiers?
- Are notes classified as contextual evidence instead of structured truth?
- Are sensitive fields restricted or excluded?
- Are unknown fields blocked until review?
- Is the folder structure safe?
- Is the data fake or safely sanitized?

A future import adapter must reject or warn on any input that fails these checks.

## Failure Modes

The contract fails if:

- Real private data is committed.
- Free text is treated as hard truth.
- Unknown fields are silently ingested.
- Restricted fields become SourceRecord fields.
- Excluded fields are copied into outputs.
- Source identifiers are missing without warnings.
- Mapping configuration is absent.
- Sanitization removes messiness so the test becomes unrealistically clean.
- The input folder encourages accidental production ingestion.
